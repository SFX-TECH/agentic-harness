The memory bank is only as good as the discipline that maintains it. Templates without this ritual rot. Run this at the close of any meaningful change, and before a session ends with work worth remembering.

Operate the three files with their distinct disciplines. Do not blur them.

## Steps

1. **decisions.md (append-only, the spine).** Did this change make a consequential, durable choice: an architecture decision, a locked trade-off, a guardrail, a vendor or schema commitment? If yes, append a dated entry now, with a one-line rationale, before it is lost. A decision banked after the fact is usually banked wrong, or not at all. Skip only if nothing consequential was decided; do not invent entries.

2. **progress.md (chronological, append-only).** Append a dated line for what actually shipped or changed. Facts, not narration: what is now true that was not true before.

3. **active-context.md (this week only, replaced not appended).** Rewrite it to the new current state. Replace stale content, do not stack on top of it. If a line no longer reflects this week, it leaves.

4. **Operational observations.** Did a reusable lesson surface, something a future session should know that is not a single decision? If it is the first time you have seen it, record it as a candidate. If it recurs something already noted, it has earned promotion to a numbered observation (use the bank-observation skill).

5. **Report** the files you touched and the one-line essence of each change, so the close is auditable.

Verify against the real files, not your memory of them: read active-context.md before replacing it and decisions.md before appending, so you preserve what is there and match its format.
